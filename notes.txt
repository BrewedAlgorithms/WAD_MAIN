--- FORM VALIDATION (lines 66-75) ---

const forms = document.querySelectorAll('.needs-validation');
forms.forEach(form => {
    form.addEventListener('submit', e => {
        if (!form.checkValidity()) {
            e.preventDefault();
            e.stopPropagation();
        }
        form.classList.add('was-validated');
    });
});

Step-by-step:

1. querySelectorAll('.needs-validation')
   Finds all <form> elements with class "needs-validation".
   This is a Bootstrap convention to mark forms needing custom validation.

2. forEach(form => ...)
   Loops through each form and attaches a "submit" event listener.

3. form.checkValidity()
   Native browser API. Checks if all fields satisfy HTML5 constraints
   (required, type="email", minlength, pattern, etc.).
   Returns true if valid, false if not.

4. e.preventDefault()
   If form is INVALID, stops form from submitting (no page reload/redirect).

5. e.stopPropagation()
   Prevents submit event from bubbling up to parent DOM elements.

6. form.classList.add('was-validated')
   Adds the Bootstrap class "was-validated" REGARDLESS of valid/invalid.
   This is the KEY Bootstrap mechanism:
   - Once added, Bootstrap CSS shows GREEN borders on valid fields
     and RED borders with error messages on invalid fields.
   - Without this class, Bootstrap shows NO validation feedback.

IN SHORT: Intercepts form submit, checks validity, blocks if invalid,
          then tells Bootstrap to visually show the validation state.


--- TOOLTIP INITIALIZATION (lines 78-81) ---

const tooltipTriggerList = document.querySelectorAll('[data-bs-toggle="tooltip"]');

Step-by-step:

1. querySelectorAll('[data-bs-toggle="tooltip"]')

SECTION 4: AJAX LOGIN REQUEST (w11/public/login.js, lines 11-36)
================================================================================

const xhr = new XMLHttpRequest();

Creates a new XMLHttpRequest instance. The browser's built-in API for making
HTTP requests without reloading the page. "xhr" represents one request cycle.


xhr.open("POST", "/login", true);

Configures the request:
  method: "POST"  -- HTTP method. POST for sending credentials to the server.
  url: "/login"   -- The backend endpoint (route in server.js).
  async: true     -- Asynchronous request. Browser doesn't freeze while waiting.


xhr.setRequestHeader("Content-Type", "application/json");

Sets an HTTP header:
  Content-Type: "application/json"
  Tells the server: "The body I'm sending is in JSON format."
  Without this, the server wouldn't know how to parse incoming data.


xhr.onload = function () {
    if (xhr.status === 200) {
        window.location.href = "data.html";
    } else {
        alert("Invalid Login");
    }
};

Callback that fires when the HTTP response is received:

  xhr.status === 200  -- HTTP 200 means "OK", login was successful.
    -> window.location.href = "data.html"  -- Redirect to protected dashboard.

  Else (401 Unauthorized, 400 Bad Request, etc.)
    -> alert("Invalid Login")  -- Login failed.


xhr.onerror = function () {
    alert("Server not running (start server.js)");
};

Fires only on NETWORK-LEVEL FAILURE (not an HTTP error status):
  - Server is unreachable
  - Server isn't running
  - DNS or connectivity issue
Different from onload with non-200 status. onerror means NO response received.


xhr.send(
    JSON.stringify({
        email: username,
        password: password,
    }),
);

Actually fires the HTTP request:

  JSON.stringify(...)  -- Converts JS object to JSON string.
  Server receives: {"email": "test@example.com", "password": "mypassword"}
  Keys "email" and "password" match what backend reads from req.body.


FULL FLOW:

  User fills form -> clicks Submit
          |
  e.preventDefault() stops normal form submission
          |
  Create XMLHttpRequest -> configure POST /login -> set JSON header
          |
  Set up callbacks (onload for response, onerror for network failure)
          |
  Send JSON payload {email, password}
          |
          |-- Status 200        -> Redirect to data.html (logged in)
          |-- Status 401/400    -> Alert "Invalid Login"
          |-- Network error     -> Alert "Server not running"

--- data-role ATTRIBUTES ---

data-role="page"
  Defines a single "page" in jQuery Mobile.
  Each page is a <div> that fills the viewport.
  Multiple pages can exist in one HTML file.
  Navigation between them uses hash links (#id).

data-role="header"
  Creates the top toolbar/bar of a page.
  Typically contains <h1>-<h6> title and optionally buttons.
  Renders as a styled fixed bar at the top.

data-role="navbar"
  Creates a horizontal navigation bar with button-like links.
  Must contain a <ul> with <li> items, each containing an <a> tag.
  Buttons are evenly spaced across the width.

data-role="content"
  The main scrollable content area of a page.
  Everything inside is the body content.
  (Deprecated in jQuery Mobile 1.4+ in favor of <main>, but still used.)

data-role="listview"
  Transforms a <ul> or <ol> into a mobile-optimized list.
  Adds dividers, borders, and touch-friendly padding.
  Can be a simple list or a "split button" list with icons.

data-role="collapsible"
  Creates an expandable/collapsible accordion section.
  The first heading element (<h1>-<h6>) becomes the clickable toggle header.
  Everything after it is the collapsible content.

data-role="footer"
  Creates the bottom toolbar/bar of a page.
  Typically contains copyright text, links, or a simple <h4> title.


--- data-theme ATTRIBUTE ---

data-theme="b"
  Applies theme swatch "b" to the element.
  jQuery Mobile uses lettered swatches (a, b, c, d, e) for color schemes.
  Swatch "b" is typically a DARK/BLACK background with WHITE text.
  Used on every header and footer for a consistent dark toolbar look.


--- data-transition ATTRIBUTE ---

Controls page transition animation when navigating between pages via <a> links.

data-transition="slide"
  New page SLIDES IN from right to left (like native mobile app navigation).
  Most commonly used transition.

data-transition="fade"
  New page FADES IN over the old page with a crossfade effect.

data-transition="flip"
  Page transition mimics a CARD FLIP animation.
  Old page flips away to reveal the new page.

data-transition="pop"
  New page POPS/SCALES IN from the center with a zoom-in effect (like a modal).

data-transition="slideup"
  New page SLIDES UP from the bottom of the screen (like a bottom sheet).


--- data-position ATTRIBUTE ---

data-position="fixed"
  Makes the footer (or header) stay FIXED/PINNED at the bottom of the viewport
  even when the user scrolls the content.
  Tapping the screen temporarily shows/hides it.
  Used on footers in w23, w24, w26.


--- CSS CLASSES ---

ui-btn-active
  Marks a navbar button/link as the CURRENTLY ACTIVE/SELECTED page.
  jQuery Mobile styles it with a highlighted/pressed appearance.
  Applied to the <a> tag that corresponds to the current page's id.

  Example pattern:
    <!-- On the "Home" page -->
    <li><a href="#home" class="ui-btn-active">Home</a></li>
    <li><a href="#about" data-transition="slide">About</a></li>

    <!-- On the "About" page -->
    <li><a href="#home">Home</a></li>
    <li><a href="#about" class="ui-btn-active" data-transition="slide">About</a></li>


--- HOW IT ALL WORKS TOGETHER ---

1. data-role="page"          wraps each logical "page" (multi-page in one file)
2. data-role="header" + data-theme="b"  gives a dark top bar with a title
3. data-role="navbar"        creates nav buttons with data-transition animations
                              and class="ui-btn-active" to highlight current page
4. data-role="content"       holds all the page content
5. data-role="listview"      stylizes lists (courses, subjects, projects)
6. data-role="collapsible"   creates accordion sections (departments, skills,
                              movie reviews, course years)
7. data-role="footer" data-theme="b" data-position="fixed"  gives a dark footer
   pinned to the bottom
